# Info-Helper Skill

Skill — это **интерактивный wizard**, который проводит пользователя через 5 этапов: знакомство → загрузка клиентов → research → отчёт → подписка на Telegram.

После первого прогона можно поставить на cron, чтобы поводы приходили каждое утро. Cron-режим описан в отдельной секции — он работает на тех же конфигах, что собирает wizard.

## Когда использовать

- Первый запуск участником воркшопа («запусти info-helper»)
- Когда пользователь хочет получать ежедневные поводы для звонков клиентам
- Когда пользователь готовится к звонку и просит «что нового у клиента X за неделю»
- Когда пользователь хочет настроить Telegram-доставку («подключи Telegram»)

## Главное правило

**Скилл всегда работает интерактивно, через диалог в чате.** Никогда не угадывает за пользователя:
- не «вот возьму первые 5 компаний из памяти»
- не «давай прогоню без Telegram, отправлю в файл»
- не «вот текст сообщения для бота — сами разберётесь»

Каждый этап = вопрос + ожидание ответа + переход к следующему этапу.

---

## Phase 1 — Знакомство (~30 сек)

Это первый этап. Без него поводы получаются обобщёнными.

Спрашивает пользователя **обе вещи** одним сообщением, ждёт ответа:

```
Привет. Я инфо-помощник по клиентам — каждое утро могу присылать тебе
свежие поводы позвонить тем, кто у тебя в воронке.

Расскажи 2 вещи, чтобы я подбирал поводы под твой контекст продажи:

1. Что вы продаёте? (Например: «AI-инструменты для отдела продаж»,
   «BI-консалтинг и дашборды», «CRM-интеграция для логистики»)

2. В какой сфере работаете? (B2B SaaS / Финтех / Логистика /
   E-commerce / Консалтинг / Маркетинг-агентство / Кибербезопасность /
   HR-tech / Другое)
```

Ждёт ответ. Парсит:
- `user_product` — строка «что продаёт»
- `user_industry` — категория из списка или «other»

Если ответ нечёткий — переспрашивает один уточняющий вопрос (не больше).

Сохраняет ответы в memory сессии — они нужны для всех следующих этапов.

## Phase 2 — Список клиентов (~1 мин)

Когда контекст известен:

```
Окей, ты продаёшь «{user_product}» в сфере «{user_industry}».
Дальше нужен список клиентов, по которым я буду собирать поводы.

Варианты:

A) Перечислить вручную — назови 5-10 компаний (просто имена, или имя + сайт)
B) Загрузить файл — прикрепи CSV/Excel из CRM
C) Пока нет своих клиентов — давай возьмём тестовый набор реальных KZ B2B компаний

Что выбираешь — A, B или C?
```

Ждёт ответ.

### Если A — пользователь перечислил компании
- Парсит ответ: разбивает по запятым / переносам строки / номерам
- Для каждого имени выделяет website если он указан (или ищет в WebSearch на следующем этапе)
- Сохраняет как `clients` в памяти сессии
- Переходит к Phase 3

### Если B — пользователь приложил файл
- Прочитай файл (CSV / Excel / Google Sheet export)
- Определи колонки: `name`, `website` обязательны; если есть `industry`, `notes`, `lpr_name` — используй
- Покажи пользователю превью первых 5 строк + общее число и спроси «всё корректно?»
- Переходит к Phase 3

### Если C — пользователь без клиентов
Прочитай `references/test-companies.yaml`. Найди подходящие компании под `user_industry` через секцию `industry_match`:

```
Хорошо. У меня есть 19 узнаваемых KZ B2B-компаний разных отраслей.
Под твою сферу «{user_industry}» подходят вот эти {N}:

  1. Sulpak (retail) — 200 магазинов техники, ТОО ARENA S
  2. Halyk Bank (banking) — системный банк
  3. Documentolog (saas) — KZ ECM/EDI провайдер
  ...

Возьмём этих? Или назови других — могу взять любые из списка, или ты
можешь добавить свои.
```

После подтверждения → Phase 3.

## Phase 3 — Research + pitch generation (~2-5 мин на компанию)

Для каждой компании из списка:

### Stage 3a — Confirm identity
WebSearch: `"{company.name}" {company.website}` — убедись, что нашёл правильную (особенно для KZ — одно имя бывает у 2-3 юр.лиц). Если уверенность <70% — спроси пользователя «уточни какая именно».

### Stage 3b — Сбор сигналов

Параллельные WebSearch'и (4 запроса):
- `"{company.name}" новости 2026`
- `"{company.name}" CEO OR директор OR назначение 2026`
- `"{company.name}" пресс-релиз OR launch OR партнёрство 2026`
- `"{company.name}" hh.kz вакансии` (для KZ-компаний)

WebFetch топ-5 ссылок. Извлеки: заголовок, дату публикации, ключевые факты (1-2 предложения), URL.

### Stage 3c — Идентификация ЛПР

Если в собранных новостях / на сайте упомянуты конкретные имена (CEO, CMO, CRO, коммерческий директор, head of sales, head of IT) — выпиши их с должностью + источник.

Если ЛПР не найден через free-стек — пометь «ЛПР не идентифицирован в публичных источниках, рекомендуется LinkedIn deep dive через `ai-camp-enrich-and-pitch` skill».

### Stage 3d — Фильтр

Для каждого факта проверь:
- **Свежесть:** ≤7 дней с даты события. Старше — drop, если только это не очень значимое событие (CEO change, M&A)
- **Конкретность:** есть имена, цифры, цитаты, конкретный продукт. Generic-факты («компания работает на рынке», «лидер индустрии») — drop
- **Actionability:** менеджер сможет сегодня использовать в разговоре с найденным ЛПР

Из оставшихся выбери top-3-5 на компанию.

### Stage 3e — Формулировка повода (под user_product!)

Это критически важный этап. Powod не должен быть generic «у них новость X». Он должен быть **pitch-shaped**: связь между новостью клиента и тем, что продаёт пользователь.

Для каждого факта сгенерируй:

**hook** (1 строка) — что произошло, конкретно:
- ❌ «Компания расширяется»
- ✅ «Открыли новый хаб в Шымкенте, площадь 3000 м², старт операций 15 мая»

**why_now** (1 строка) — почему сейчас это создаёт окно для нашего pitch:
- Связь между событием и тем, что продаёт пользователь
- ❌ «это интересный сигнал»
- ✅ «новый хаб = 30+ операторов + WMS + новые маршруты — точка где наш {user_product} даёт быстрый ROI»

**lpr_target** — кому звонить (если ЛПР идентифицирован):
- ✅ «Айбек Каримов, коммерческий директор»
- если ЛПР не найден: «коммерческому директору или Head of Sales — конкретное имя нужно искать в LinkedIn»

**opening_line** — готовая первая фраза менеджера:
- max 20 слов, русский, деловой на «вы»
- ссылка на событие + открытый вопрос
- KZ-этикет где уместно (по имени-отчеству если знаешь)
- без штампов («здравствуйте, я из ...», «уважаемая компания»)

**source** — URL события.

Пример:
```
🎯 Sulpak открыл хаб в Шымкенте (3000 м², 15 мая)
   📅 19 мая (2 дня назад)
   🎯 ЛПР: Дмитрий Провкин, ген. директор · LinkedIn
   💡 Почему сейчас: новые маршруты + рост WMS-нагрузки = окно для нашего
      «AI-инструменты для отдела продаж» — приоритизация лидов из нового региона
   💬 «Дмитрий Александрович, видел про хаб в Шымкенте — у sales-команды
       в новом регионе обычно проседает приоритизация лидов. Расскажу за 5 минут?»
   🔗 sulpak.kz/press/shymkent-hub
```

## Phase 4 — Markdown отчёт + показ

После того как все компании обработаны:

1. **Собери отчёт** `reports/info-helper-<YYYY-MM-DD>.md`:
   ```markdown
   # Поводы касания на <today_ru>

   > {N} поводов по {M} клиентам. Продаём: «{user_product}». Сфера: «{user_industry}».

   ---

   ## {company.name}

   ### Повод 1
   🎯 {hook}
   📅 {date}
   🎯 **ЛПР:** {lpr_target}
   💡 **Почему сейчас:** {why_now}
   💬 «{opening_line}»
   🔗 {source}

   (повторить 3-5 раз)

   ---
   ```

2. **Открой в браузере** (или встроенный viewer):
   ```bash
   open reports/info-helper-<date>.md
   ```

3. **Покажи пользователю в чате саммари:**
   ```
   Готово. Собрал {N} поводов по {M} клиентам.
   Отчёт открыт: reports/info-helper-{date}.md

   Топ-3 по моему мнению:
   1. {hook 1} → {lpr_target 1}
   2. {hook 2} → {lpr_target 2}
   3. {hook 3} → {lpr_target 3}

   Дальше — Phase 5: подписка на Telegram, чтобы это приходило каждое утро.
   ```

## Phase 5 — Telegram-подписка (~3 мин)

После того как пользователь увидел первый отчёт — предложи подписку:

```
Этот отчёт собирается за 2-5 минут, но каждый день вручную — лень.
Хочешь чтобы поводы приходили в Telegram каждое утро в 9:00?

Если да — пройдём настройку за 3 минуты, я веду пошагово.
Если нет — отчёты будут сохраняться в reports/ как Markdown,
                                                                 Telegram можно подключить позже («настрой Telegram»).
```

### Если да

Следуй пошагово инструкции из `references/telegram-setup-wizard.md`:

1. **Шаг 1 — создать бота** через @BotFather, скопировать токен
2. **Шаг 2 — узнать chat_id** через бота + getUpdates URL
3. **Шаг 3 — тестовая отправка** через `curl`, проверить что пришло
4. **Шаг 4 — сохранить настройки** в `.env` + `config/settings.yaml`
5. **Шаг 5 — установить cron / launchd** (3 варианта на выбор)

Каждый шаг — спрашиваешь пользователя в чате, ждёшь ввод, валидируешь, переходишь к следующему. **Не просишь всё сразу одним блоком.**

### Если нет

Сохрани `config/clients.yaml` (если ещё не сохранил — из тех клиентов что собрали в Phase 2). В `config/settings.yaml` оставь `delivery.telegram.enabled: false`. Скажи:

```
Хорошо. Отчёты будут сохраняться в reports/.
Telegram можно подключить позже — скажи «настрой Telegram», я проведу через шаги.

Чтобы прогнать инфо-помощник ещё раз руками:
  /info-helper

Или если хочешь автоматический cron без Telegram (отчёт только в файл):
  ./runner/run.sh --once
```

## Что после wizard — Cron режим

После завершения Phase 5 скилл может быть запущен **headless** через `runner/run.sh` (cron / launchd / systemd). В этом режиме:

- Читает `config/clients.yaml` и `config/settings.yaml` — то, что собрал wizard
- НЕ задаёт вопросов — при неуверенности пропускает компанию и логирует
- Прогоняет Phase 3 (research + pitch) для всех клиентов
- Сохраняет отчёт + (если включено) шлёт в Telegram
- Идемпотентен в течение дня — `cache/sent-events.json` хэширует уже отправленные события за 14 дней

Подробности cron-режима — в [runner/README.md](../../runner/README.md).

---

## Inputs (формальные)

### Интерактивный режим
Никаких inputs кроме slash-command `/info-helper` или фразы-триггера. Всё остальное собирается через диалог.

### Cron-режим (после первого wizard)
- `config/clients.yaml` — список клиентов + контекст продаж (создан в Phase 2-3 wizard)
- `config/settings.yaml` — настройки + Telegram (создан в Phase 5)
- `cache/sent-events.json` — кэш отправленных событий (автоматически)

## Free vs Paid stack

| Источник | Free | Paid |
|----------|------|------|
| Web search | ✅ WebSearch (built-in) | — |
| Page content | ✅ WebFetch (built-in) | — |
| KZ company facts | ✅ adata.kz через WebFetch | goszakup API (нужен token) |
| LinkedIn posts | ❌ | Apify `harvestapi/linkedin-profile-scraper` (`APIFY_API_KEY`) |
| Twitter/X | ❌ | xAI Grok через `x-research` skill (`XAI_API_KEY`) |
| HH.kz vacancies | ✅ WebFetch на vacancy URL | — |

Скилл работает out-of-the-box на free-стеке. Paid интеграции автоопределяются по env-переменным и добавляют точности.

## Зависимости в репо

- `references/test-companies.yaml` — тестовые KZ B2B-компании для Phase 2 (вариант C)
- `references/telegram-setup-wizard.md` — пошаговый wizard для Phase 5

- `runner/run.sh` — headless-обёртка для cron-режима
- `delivery/telegram.sh` — shell fallback для Telegram (используется внутри Phase 5 для тестовой отправки)

---

## Что важно для wizard-режима

1. **Никаких bulk-вопросов** — один этап = один вопрос (максимум парный, как в Phase 1)
2. **Не пропускай Phase 1** даже если контекст «вроде понятен» — он нужен для pitch-формулировок
3. **В Phase 5 никогда не вставляй токены и chat_id в код** — только в `.env`, который в `.gitignore`
4. **Тестовая отправка обязательна** перед сохранением cron — иначе пользователь узнает что не работает только завтра в 9:00
5. **Не уходи в headless** без явного запроса пользователя — wizard не должен «втихую» поставить cron

## Edge cases

### Пользователь сказал «у меня нет клиентов и тестовые мне не интересны»
Покажи список реальных публичных KZ B2B компаний и попроси выбрать любые 3-5: «возьмём как примеры, чтобы ты увидел как работает; для своих клиентов прогон будет такой же».

### У пользователя 100+ клиентов в CSV
Не прогоняй все. Скажи «прогон на 100+ займёт час — давай для демо возьмём 5-10, остальное cron сделает фоном завтра». Спроси какие 5-10.

### Пользователь спросил «а почему 3-5 поводов, я хочу 10»
Объясни: 10 — это шум, менеджер не успеет отработать. 3-5 — это TOP-фактов с самым высоким actionability. Остальное запушено в Phase 3d-фильтре.

### Свежих поводов по компании не нашлось
Не выдумывай. В отчёте напиши «{company.name}: свежих поводов за 7 дней не нашлось. Попробуй завтра или расширь окно до 14 дней.»

### Пользователь хочет slack/email вместо Telegram
В Phase 5 предложи: «Telegram сейчас (быстрее всех), slack/email — через `config/settings.yaml` после первого прогона». Файл уже имеет placeholder секции для slack и email.
